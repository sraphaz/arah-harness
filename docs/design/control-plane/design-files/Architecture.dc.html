Architecture

Two repositories, one contract: kernel and consumer

The harness is developed and versioned in its own repository (the product), and installed into consumer repositories as a maintained kernel. Everything the consumer runs is versioned, inspectable and regenerable.

Product repository and consumer repository

The kernel (agents, skills, scripts, schemas, templates) is developed in arah-harness and distributed into consumers by install and regenerate. The consumer keeps its own state: config, signals, audit, organism.

Contracts, storage and generation

Every mechanism is a file with a schema — no hidden state, no external database.

CONTRACTS
Schema-validated YAML
Agent manifests, skill manifests, choreography, domains and config validate against schemas/ on every gate run.
SIGNAL STORAGE
Append-only JSONL bus
Typed events under .arah/bus/ — readable by any tool, diffable in PRs, portable across models.
AUDIT STORAGE
Session ledger
Actions, results, autonomy decisions and session telemetry under .arah/audit/.
GRAPH GENERATION
Derived, never hand-written
arah export-graph derives the agent graph from manifests; a gate fails if the graph drifts from its sources.
IDE INTEGRATION
Rules + ARAH Live
Installed .cursor/ rules and AGENTS.md bind Cursor, Claude Code and Copilot to the contracts; ARAH Live (experimental) streams bus events into the IDE.
CI INTEGRATION
Gates in workflows
The same gates run in .github/workflows/: a PR that fails validation cannot be merged.

Update lifecycle

The consumer stays consistent with the kernel through regeneration — never by manual copying.

1
kernel update
new agents, skills, gates land in arah-harness
→
2
arah regenerate
kernel synced, discovery re-run, domains synced
→
3
arah doctor · sync-check
installation validated, drift detected
→
4
pull request
the update itself is reviewed and merged by humans

Every agent needs a role, a scope, a capability and an accountable path to production.
