A transformação

A engenharia de software está se tornando engenharia humano-agente

O trabalho deixou de ser apenas escrever código. É coordenar humanos, agentes, modelos, ferramentas, políticas, contexto, validação e memória persistente — como um único sistema de engenharia.

O futuro da engenharia de software não é autônomo. É coordenado.

O problema

Gerar código não é o problema mais difícil

O mais difícil é coordenar inteligência sem perder contexto, autoridade e responsabilidade.

P·01
Agentes sem contexto organizacional
Modelos leem código localmente, mas não conhecem responsabilidades, arquitetura, fronteiras de domínio nem critérios de decisão.
P·02
Prompts efêmeros
O conhecimento produzido em sessões de IA desaparece quando a conversa termina ou a ferramenta muda.
P·03
Autonomia sem responsabilidade
Agentes executam mudanças sem autoridade explícita, escopo, caminho de aprovação ou evidência.
P·04
Conhecimento preso a fornecedores
Contexto e workflows ficam presos em copilotos, plataformas proprietárias e runtimes específicos.
P·05
Políticas apenas documentais
Regras vivem em wikis e PDFs, sem forma executável que um agente seja obrigado a obedecer.
P·06
Multiagente sem coordenação
Múltiplos agentes atuam no mesmo repositório sem papéis, fronteiras ou sinais compartilhados.
P·07
Ausência de evidências
Mudanças chegam sem artefatos que provem o que foi validado, por quem, sob qual política.
P·08
Contexto perdido entre sessões
Cada nova sessão começa do zero: decisões, restrições e histórico precisam ser reexplicados.
A inversão

O repositório governa o agente.

O agente não recebe autoridade irrestrita. O repositório fornece papel, contexto, escopo, capacidades, políticas, validações e caminhos de aprovação.

Agent-first
Prompt→ Agente→ Ferramentas→ Mudança de código

A autoridade nasce da conversa. O contexto vive na sessão. Nada persiste.

Repository-first · ARAH
Intenção humana→ Contexto do repo→ Domínio→ Agentes→ Skills→ Gates→ Evidências→ Pull request

Autoridade, escopo e validação são contratos versionados no repositório. Tudo persiste.

O que o ARAH não é

O ARAH é uma camada repo-first de governança e coordenação da engenharia entre humanos e agentes. Ele não é:

Autonomia sem governança não escala.
✕Um modelo de linguagem
✕Um chatbot
✕Um copiloto de código
✕Apenas um framework de prompts
✕Apenas um runtime multiagente
✕Uma plataforma de CI/CD
✕Um substituto do GitHub
✕Auto-merge irrestrito
✕Uma promessa de desenvolvimento de software sem humanos

Início rápido

Instale em um repositório existente

Três comandos para instalar o kernel e uma superfície de CLI enxuta para descobrir, sincronizar, validar e evoluir. Guia completo na documentação.

arah doctor
arah discover
arah domain sync
arah export-graph
arah evolve
1 · CLONAR O HARNESS
git clone https://github.com/sraphaz/arah-harness.git $env:USERPROFILE\arah-harness
2 · INSTALAR NO SEU PROJETO
cd C:\path\to\your-project

powershell -ExecutionPolicy Bypass `
  -File $env:USERPROFILE\arah-harness\cli\arah.ps1 `
  install `
  -ProjectName "my-project"
3 · REGENERAR / ATUALIZAR
powershell -File `
  $env:USERPROFILE\arah-harness\cli\arah.ps1 `
  regenerate `
  -Target . `
  -UpdateKernel

Agentes propõem. Humanos selecionam. O repositório lembra.

Seu contexto de engenharia deve pertencer ao seu repositório, não a um fornecedor de modelo.
